FHIRBridge Developer Handover Guide
A beginner-friendly explanation for a developer who is new to this project, including architecture, endpoints, methods, data flow, deployment, and debugging.
1. What This Project Is
FHIRBridge is a small healthcare interoperability application. It accepts patient data and HL7 messages, converts them into FHIR-style JSON, stores records in a database, and can send FHIR resources to a FHIR server endpoint.
A developer should think of it as five cooperating parts: React frontend, FastAPI backend, worker process, Postgres database, and RabbitMQ message queue.
    • Frontend: the browser interface for viewing logs, loading FHIR resources, editing JSON, and submitting to FHIR endpoints.
    • Backend: the HTTP API. It validates requests, stores data, publishes HL7 jobs, and submits FHIR resources.
    • Worker: background process that consumes HL7 jobs from RabbitMQ and converts them to FHIR JSON.
    • Postgres: persistent storage for patient and HL7 records.
    • RabbitMQ: queue used to pass HL7 messages from backend to worker.

6. Backend Application Startup
backend/app/main.py creates the FastAPI application. The key startup steps are:
    1. lifespan(app) calls init_db() when the API starts.
    2. init_db() creates SQLAlchemy tables and adds missing JSONB columns if needed.
    3. FastAPI CORS middleware allows the local frontend on ports 3000 and 5173.
    4. Routers are included for health, dashboard, FHIR lookup, FHIR submit, HL7, and patients.
This means backend/app/main.py is the best first file to open when trying to understand how API routes are wired into the server.

8. Main Data Flows
8.1 JSON Patient Flow
    5. Client sends POST /api/patient with external_id, name, dob, gender, phone, and address.
    6. accept_patient() receives a PatientCreate model.
    7. create_patient() calls convert_patient_to_fhir().
    8. convert_patient_to_fhir() calls convert_json_patient_to_fhir().
    9. The result is a FHIR Patient resource with resourceType Patient.
    10. auto_submit_fhir_resource() sends the resource to configured endpoints when auto-submit is enabled.
    11. A PatientRecord row is saved in Postgres with fhir_json and fhir_submission_results.
8.2 HL7 Message Flow
    12. Client sends POST /hl7 with raw text/plain HL7 message.
    13. receive_hl7_message() calls queue_hl7_message().
    14. queue_hl7_message() checks the message is not empty and starts with MSH|.
    15. create_queued_hl7_record() saves a queued HL7Record row in Postgres.
    16. publish_message() sends a job to RabbitMQ queue hl7_messages.
    17. worker/app/main.py starts consume_hl7_messages(process_hl7_message).
    18. process_hl7_message() converts HL7 to FHIR Bundle and updates the HL7Record to processed.
    19. auto_submit_fhir_resource() may submit the generated Bundle to configured FHIR endpoints.
8.3 Dashboard Flow
    20. Frontend calls GET /api/patient-records and GET /api/dashboard-events together.
    21. fetch_patient_records() returns summary rows from patients table.
    22. fetch_dashboard_events() combines patient records and HL7 records into one timeline.
    23. Frontend shows metrics, logs, processing status, and FHIR submit status.
8.4 Manual FHIR Submit Flow
    24. Developer opens the Endpoint Submit tab in the frontend.
    25. Frontend calls GET /api/fhir/{resource_id} to load stored FHIR JSON.
    26. Developer can edit the JSON in the textarea.
    27. Frontend sends POST /api/fhir/submit with resource_id, endpoints, mode, and optional edited resource.
    28. submit_fhir() calls submit_fhir_resource().
    29. submit_fhir_resource() builds the correct FHIR URL and sends HTTP requests to each endpoint.

10. FHIR Mapping Logic
The mapper converts ordinary app data or HL7 segments into FHIR JSON. This is the heart of the interoperability behavior.
Method
Input
Output / job
split_components(value)
HL7 field value like DOE^JOHN
Splits by caret into HL7 components.
hl7_date_to_fhir(value)
HL7 date like 19900101
FHIR date string like 1990-01-01.
parse_hl7_message(hl7_message)
Raw HL7 text
Dictionary grouped by segment name, such as MSH, PID, PV1.
parse_message_type(segments)
Parsed HL7 segments
Message type and trigger event, such as ADT^A01 and A01.
build_patient_identifier(raw_identifier, fallback_id)
PID-3 value
FHIR identifier list and chosen patient ID.
build_human_name(raw_name)
PID-5 value
FHIR HumanName list.
build_address(raw_address)
PID-11 value
FHIR Address list.
build_telecom(values)
PID phone fields
FHIR telecom list.
convert_json_patient_to_fhir(patient)
PatientCreate JSON model
FHIR Patient resource.
convert_hl7_adt_to_fhir_bundle(message_id, hl7_message)
Message ID plus raw HL7
FHIR Bundle with MessageHeader, Patient, and optional Encounter.

Important mapping constants:
    • HL7_GENDER_MAP maps M, F, O, U, A, and N into FHIR gender values.
    • ADT_ENCOUNTER_STATUS maps common ADT trigger events to FHIR Encounter statuses.
    • The HL7 Bundle includes MessageHeader and Patient. If PV1 exists, it also includes Encounter.
11. FHIR Submit Logic
FHIR submit is handled by backend/app/services/fhir_submit_service.py. It supports two modes.
Mode
URL pattern
HTTP method
Meaning
validate
{base}/{resourceType}/$validate
POST
Ask the FHIR server if the resource is valid without storing it.
submit with id
{base}/{resourceType}/{id}
PUT
Create or update a specific resource ID.
submit without id
{base}/{resourceType}
POST
Ask the FHIR server to create a new resource ID.

The service also handles a duplicate resource case. If the server returns status 412 and the response says a resource already exists, duplicate_resource_id_from_outcome() extracts the existing ID and retries with PUT.

17. Debugging Guide
Symptom
What to check
Frontend does not load
Run docker compose ps. Check frontend is Up and port 5173 is published.
Frontend loads but data is empty
Check backend logs and open /api/patient-records.
Backend /api/health gives 404
Use /health. The health route is not under /api.
HL7 does not process
Check RabbitMQ is healthy, worker is Up, and worker logs show processed message.
FHIR submit fails
Check endpoint URL, resourceType, resource id, and response body in submit results.
Database errors
Check postgres container health and DATABASE_URL.
RabbitMQ connection errors
Check RABBITMQ_URL and rabbitmq container health.
Docker permission errors
Start Docker Desktop and run commands from a terminal that can access Docker.

Useful commands:
docker compose ps
docker compose logs -f backend
docker compose logs -f worker
docker compose logs -f rabbitmq
docker compose up -d --build
18. Production Checklist
    • Create a real .env file. Do not use .env.example for real production secrets.
    • Change Postgres and RabbitMQ default credentials.
    • Do not expose database port 5432 or RabbitMQ port 5672 publicly.
    • Put frontend and backend behind HTTPS.
    • Build frontend static files for production instead of running Vite dev server.
    • Decide whether FHIR_AUTO_SUBMIT_ENABLED should be true. For safety, many environments should validate first, then submit manually.
    • Add authentication before using this with real patient data.
    • Add audit logging and stronger error handling before using protected health information.
